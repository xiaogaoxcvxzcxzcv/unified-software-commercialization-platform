# ADR-0001：模块化单体与技术基线

- 状态：accepted
- 日期：2026-07-12

## 背景

平台需要服务多款软件并长期扩展，但当前团队规模和流量不需要微服务带来的独立部署复杂度。AI 可以加快编码，却也会放大边界不清和重复实现的问题。

## 决策

- 后端采用 Go 模块化单体。
- 管理后台采用 React + TypeScript。
- 事实数据使用 PostgreSQL；Redis 和 S3 兼容存储按能力接入。
- 外部接口使用版本化 REST + OpenAPI。
- 模块在代码、数据访问和契约层面隔离，但初期作为一个后端进程部署。

## 后果

- 本地和初期生产部署简单，事务边界清楚。
- 必须严格禁止跨模块 Repository 和数据表访问，否则模块化只剩目录形式。
- 未来只有在负载、团队边界或可靠性要求提供证据时才拆服务。

## 备选方案

- 继续 PHP 单文件后端：否决，难以支撑清楚的多产品模块边界。
- 立即微服务化：否决，当前运维和一致性成本高于收益。
