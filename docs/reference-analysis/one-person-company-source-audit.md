## 禁止照搬

### 模型不适合统一软件后台

- 参考项目的 Tenant 表示“不同客户站点”，我们的 Product 表示“不同软件”。二者不是同一概念。
- 参考项目把 `membershipTier` 和到期时间直接放在 User 上；我们的权限必须是 `user_id + product_id + entitlement`。
- 参考项目每个租户拥有独立用户；我们的目标是同一个账号登录多款自有软件。

### 多租户隔离不完整

- 项目写了统一租户 Prisma 代理，但 API 路由没有实际统一使用。
- 72 个 API 路由中有大量直接 Prisma 调用，依赖人工添加 tenantId。
- 套餐按 ID 更新和删除时存在未校验租户归属的路径，不能作为安全模板。

### 可靠性和并发问题

- 兑换码先更新用户再更新兑换码，没有放在同一个数据库事务中。
- 兑换码使用 `Math.random()` 生成，Schema 没有租户范围内唯一约束。
- Token 黑名单、限流、扫码登录状态和租户缓存使用进程内 Map；重启、多实例和 Serverless 环境会丢失状态。
- 支付、订单和权益没有通过清晰模块与 Outbox 解耦。

### 密钥处理问题

- 存储、微信和支付凭据保存在 JSON 字段中。
- 管理 API 会把这些配置对象返回前端，未发现字段级加密和只写不回显策略。
- 新平台必须使用密钥管理或加密存储，敏感值保存后只返回掩码。

### 工程问题

- 无自动化测试、锁文件和可复现部署说明。
- 多个 `.bak`、`.new` 和 `before_v` 文件混入源码。
- `strict: false`，多租户改造中的类型错误直到生产构建才暴露。

## 对新平台的转化规则

| 参考能力 | 新平台正确落点 |
|---|---|
| 会员套餐 | catalog + entitlement，不写入 User |
| 微信支付 | payment Provider，只发布支付事实 |
| 订单支付后开会员 | order.completed 事件驱动 entitlement.grant |
| 兑换码 | license 事务兑换，再调用 entitlement |
| 租户隔离 | 先验证 ProductContext，再由服务端解析产品下属 TenantContext |
| 进程内扫码状态 | Redis 或持久化挑战记录，带过期和一次性消费 |
| JSON 密钥配置 | 密钥引用、加密存储和后台掩码显示 |
| 通用 CRUD 页面 | Feature Block + API Client，不直接访问 Service/DB |
| 分销与提现 | P3 可选模块，不进入第一版核心 |

## 获取“完整可重建源码”时的要求

不需要真实密钥和生产数据，但必须包含依赖锁、完整迁移、`.env.example`、构建部署说明、测试和当前线上版本对应的源码，并证明以下命令通过：

```text
npm install
prisma generate
npm run build
```
