# 统一后台产品范围

本文档是“我们到底要做什么”的长期记忆。聊天中的新想法必须先与本文档对照，再决定进入当前阶段、后续阶段或暂不实施。

## 一句话目标

为所有自有桌面软件提供同一套商业化基础设施：账号统一、产品隔离、权益统一判断、支付统一接入、版本和配置统一管理，并通过 SDK 让新软件低成本接入。

## 四个核心概念

### 用户 User

代表同一个真实账号。用户可以登录多款软件，账号资料和安全状态全局共享。

### 产品 Product

代表一款独立软件。套餐、权益、设备、订单、版本、配置、文件和用量都按 `product_id` 隔离。

### 代理租户 Product Tenant

租户是某一款软件下面的代理经营和数据隔离功能，不是统一平台的中心。每个产品自动拥有一个 `official` 官方租户，并可以创建多个 `agent` 代理租户。

租户不能脱离产品存在，技术范围始终是 `product_id + tenant_id`。同一个代理如果经营两款软件，会在两款产品下分别拥有代理租户。

### 权益 Entitlement

代表“某用户在某产品和代理租户中可以使用什么、到什么时候、允许多少设备”。付款、激活码、试用、赠送和人工开通最终都转换为权益。

这四者不能合并。特别禁止把“会员等级”直接写成用户的全局字段，也禁止建立脱离产品的顶层租户中心。

## 统一能力的实现方式

公共能力只在统一后台实现一套，再按产品启用和配置，不把后台代码复制进每款软件。

```text
统一后台能力库
├─ 登录
├─ 权益
├─ 激活码
├─ 代理租户
├─ 订单支付
├─ 版本更新
├─ 远程配置
├─ 云存储
└─ API 用量

产品 A -> 开启需要的能力
产品 B -> 开启需要的能力
未来产品 -> 直接接入已有能力
```

每款软件内部只保留自己的核心业务、界面、统一 SDK、必要的离线授权缓存和更新程序。

## P0：第一版必须完成

| 能力 | 必须做到的结果 |
|---|---|
| 产品中心 | 创建产品、产品状态、客户端身份和产品上下文 |
| 多端应用上下文 | 同一软件管理桌面、Web、H5、App 和微信小程序的独立凭据、登录/支付渠道与安全回跳 |
| 产品能力配置 | 为每款软件启用或停用公共能力，生成对应后台目录 |
| 产品内代理租户 | 每个产品有官方租户，可创建、停用代理租户并隔离数据 |
| 统一账号 | 注册、密码/微信登录、外部身份绑定与合并、刷新、退出、冻结、密码修改 |
| 权益中心 | 查询、授予、延长、撤销、来源流水和到期判断 |
| 设备管理 | 绑定、解绑、设备上限、撤销和最后活跃时间 |
| 激活码 | 批量生成、兑换、过期、批次、幂等和产品隔离 |
| 管理后台 | 登录、产品切换、用户查询、权益和设备操作 |
| 客户端 SDK | 初始化、登录、检查权益、设备绑定、错误与重试 |
| 用户前台基础 | Hosted/Embedded 登录、个人中心、当前权益、设备和激活码页面契约 |
| 安全审计 | 管理员操作、登录失败、权限拒绝和敏感操作记录 |
| 工程基础 | 数据库迁移、健康检查、配置、日志、CI 和备份框架 |

P0 验收标准：产品 A 不能读取产品 B 的业务数据；同一产品内代理 A1 不能读取代理 A2 的数据；第一款真实桌面软件可以通过 SDK 使用后台为它启用的公共能力。

## P1：形成商业闭环

| 能力 | 必须做到的结果 |
|---|---|
| 商品与套餐 | 免费、试用、月付、年付、永久和功能项 |
| 订单中心 | 服务端定价、商品快照、订单状态和取消超时 |
| 微信支付 | 下单、二维码、验签回调、主动查询和重复通知幂等 |
| 退款与对账 | 退款状态、权益回收、日对账和异常订单处理 |
| 版本更新 | 版本检测、安装包哈希、灰度、强制更新和回滚 |
| 远程配置 | 公告、客服二维码、帮助链接和功能开关 |
| 通知 | 支付、到期、异常和管理员告警 |
| 代理基础经营 | 代理管理员、销售归属、代理状态和订单归属 |
| AI Gateway | 统一模型目录、Provider 适配、逻辑模型路由、流式调用、限流与故障转移 |
| AI 用量计费 | 管理员配置成本价/销售价/代理价，支持额度预占、真实用量结算、流水与对账 |

P1 验收标准：支付成功只产生一次订单完成和一次权益授予；支付失败、重复回调、退款和网络中断都有可恢复路径。

## P2：规模化运营

| 能力 | 说明 |
|---|---|
| 文件与云同步 | 安装包、公共资源、用户私有文件、配额和签名 URL |
| 数据看板 | 用户、收入、转化、版本、成本和产品对比 |
| 优惠券 | 折扣、有效期、使用上限和产品范围 |
| 开票 | 开票申请、状态和订单关联 |
| 多管理员角色 | 超管、客服、财务、运营和最小权限 |
| 开放 API | 第三方接入凭据、范围、配额和审计 |

## P3：确认有业务需求后再做

- 分销推广、佣金结算和提现。
- 代理白标站点、自定义域名和代理独立支付配置。
- 短信、邮件和复杂营销自动化。
- 团队账号、企业组织和席位计费。
- 海外支付、多币种和税务能力。

P3 不得为了“参考项目里有”就提前开发。进入 P3 必须有真实业务场景、数据模型评审和单独 ADR。

## 独立轨道：私有部署

私有部署的实例、签名许可证、离线宽限、升级兼容、备份恢复和诊断能力使用独立 Deployment 模块，不复用代理 Tenant 或普通用户激活码。架构和许可证格式提前固定，完整实现由第一位真实私有部署客户驱动。

## 软件接入标准

执行级接入步骤、AI 修改边界和兼容验收见 `software-integration-standard.md`。

每款软件只需要配置自己的产品身份并使用统一 SDK：

```text
initialize(product credentials, app version, distribution context)
login()
check_entitlement()
bind_device()
redeem_license()
check_update()
get_remote_config()
report_usage()
```

客户端不能持有支付密钥、数据库密码、管理员权限或最终授权判断逻辑。产品和租户上下文必须由服务端验证，不能相信客户端随意提交的 `product_id` 或 `tenant_id`。

## 明确非目标

- 第一版不做微服务。
- 不复制旧 AI 工具箱、一人公司项目或 Sub2API 的核心代码。
- 不把管理后台页面直接连接数据库。
- 不把 Redis 或内存 Map 当作订单、会话撤销、扫码状态或支付事实的唯一存储。
- 不以“页面能打开”代替接口、安全、备份和恢复验收。

## 决策原则

1. 先保证账号、产品和权益模型正确，再增加支付和运营功能。
2. 一个业务事实只有一个权威模块。
3. 所有跨产品数据访问默认拒绝。
4. 同一产品内的跨代理租户访问默认拒绝。
5. 所有外部回调默认会重复、延迟、乱序或失败。
6. 所有桌面客户端默认可以被逆向，不能信任客户端秘密和判断。
7. 功能按阶段闭环交付，不在最后一次性检查。
