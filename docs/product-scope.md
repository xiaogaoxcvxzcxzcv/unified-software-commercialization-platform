# 可装配软件通用能力底座：产品范围与核心目标

本文档是“我们到底要做什么”的最高产品真相。路线图、工程地图、能力目录、界面、契约和代码都必须服从本文；聊天中的新想法先与本文对照，不允许再把“统一管理后台”误写成项目的全部或中心。

## 唯一中心目标

建设一套可持续扩展的软件通用能力底座：创建新软件时选择目标端、完整能力包和用户前台 UI 模板，平台即装配可运行的共享后端、唯一统一管理后台、可维护的用户前台源码、SDK、配置、测试与说明；开发者只需继续开发该软件独有的业务。

第一目标是减少以后开发新软件的重复工作。统一管理后台是所有能力包共用的运营管理界面，是重要交付面，但不是产品中心。

用户前台本身也是平台可装配、可生成的正式交付物，不是留给开发者从零搭建的空壳。统一管理后台的软件创建和受控变更流程允许为每个目标端选择一套兼容 UI 模板；一套可交付模板必须提供可运行的前台框架、布局、导航、主题、公共页面编排和扩展槽，并完整呈现它声明支持的登录、注册、个人中心、会员权益、套餐购买等公共 Feature Block。公共页面是否生成和显示由蓝图中所选的 `available` 完整能力包决定；没有选择对应能力包时不得生成假页面、空入口或不可用按钮。

平台不生成一套适用于所有软件的业务首页、业务目录页、工作台或“中间内容”。装配完成后，这些页面以及软件的核心功能、数据和流程由该软件自己的独立开发任务在 custom/Extension 范围实现，再通过登记的路由、导航、槽位、SDK/API 和事件接入。换句话说，平台生成完整可运行的前台基础与所选公共功能，开发人员只在该软件项目内补真正独有的业务内容。

平台对一款新软件的标准交付终点是：所选公共能力已经在用户前台、统一管理后台、共享后端和 SDK/API 中真实可用，生成源码可以安装启动，Manifest/lock、测试证据和软件本地开发交接说明齐全。到达这个终点后，该软件的正文内容、业务目录、工作台和核心流程转入该软件自己的独立开发任务，不再属于统一底座的装配任务。统一底座后续只通过公开契约提供公共能力升级、兼容、回滚和扩展支持。

开发计划中用于证明装配链的第一款、第二款和第三款软件，统一称为“装配验收软件 A/B/C”。它们只验证创建是否成功、所选功能是否同时出现在正确的用户前台和统一后台、UI/SDK/API 是否可用、产品是否隔离，以及重生成和升级是否安全；除用于验证扩展边界的最小夹具外，不开发这些软件的真实正文业务。

## 用户最终获得什么

```text
创建软件
-> 选择 Web / 桌面 / H5 / 微信小程序 / 手机 App
-> 勾选已经完成的通用能力包
-> 选择与目标端兼容的用户前台 UI 模板
-> 预览依赖、配置、源码和测试交付物
-> 创建 Product、Application、official Tenant 和环境凭据
-> 启用共享后端与统一管理后台功能
-> 生成用户前台组合源码、接入壳、配置和锁定清单
-> 运行接入与隔离测试
-> 生成该软件本地的 AI/开发人员交接说明
-> 将结果移交给该软件自己的独立开发任务
-> 只在该软件项目内开发独有业务
```

例如创建图片处理软件并选择账号、权益、设备、激活码和会员购买后，软件应直接具备登录、个人中心、会员状态、我的设备、激活码兑换、套餐购买和订单等用户前台；统一后台应直接具备对应管理页面；真实后端、SDK、配置、错误恢复、测试和说明也必须同时存在。开发者只继续开发画布、图层、素材和图片任务。

## 最小产品交付单位：完整能力包

创建软件时勾选的是 `package_id` 表示的完整能力包，不是一个菜单、一个开关或某条底层 API。每个可勾选能力包必须同时交付：

1. **产品结果**：最终用户能做什么、管理员能管理什么、依赖哪些包、支持哪些端。
2. **真实后端**：业务规则、数据库迁移、API/事件、权限、隔离、审计、幂等、重试和恢复。
3. **统一管理后台**：真实 API Client、菜单注册、页面、权限和审计入口。
4. **用户前台**：页面或组件、完整状态、目标端适配以及可选择的 UI 模板。
5. **SDK 与公开 API**：稳定方法、类型、错误、超时、取消、重试和渠道适配。
6. **配置与 Provider**：配置 Schema、默认值、环境要求、密钥要求和启用前检查。
7. **源码交付**：可阅读、可维护的前台组件/模板、接入壳、示例和 SDK 源码；共享后端源码保留在平台仓库集中维护。
8. **质量证据**：单元、集成、契约、UI、端到端、隔离、失败恢复、升级和回滚测试。
9. **使用说明**：接入、配置、扩展、排错、升级、废弃和数据保留说明。

确实不适用的交付面必须明确写 `N/A` 和原因。缺少任一必需交付面、仍使用演示数据或只有页面外壳的功能，不得出现在普通创建流程的可勾选目录。

完整定义见 `complete-capability-package-standard.md`，可勾选包及真实状态见 `capability-package-catalog.md`。`capability-index.md` 继续登记底层原子能力，两者不能混为一张表。

## 产品组成

### 通用能力底座负责

- 软件创建、目标端选择、能力包选择、依赖解析和装配结果。
- 账号、用户中心、权限、权益、设备、激活码、商品、订单、支付和退款。
- 版本更新、远程配置、通知、文件存储、AI 调用、额度、用量和 API Key。
- 产品内代理租户、代理管理、审计、运营统计和私有部署许可等可选能力。
- 唯一统一管理后台、用户前台模板、Hosted UI、SDK、公开 API 和接入测试。
- 能力包、模板、SDK 和生成结果的版本、升级、兼容与回滚。
- 为生成软件交付根目录级开发规则、源码所有权、扩展点、公共能力调用方法和启动/测试命令，使后续 AI 不依赖聊天记录即可继续开发。

### 每款软件自己负责

- 软件独有的业务首页、业务目录页、工作台、页面、工作流、数据结构和业务规则。
- 例如视频项目、素材、剪辑和生成任务；文章、模板和生成记录；图片画布、图层和处理任务。
- 产品名称、Logo、品牌素材、主题 Token、合规文案和独有权益映射。
- 在平台定义的扩展点挂载软件独有的前台和后台入口。

当某项独有功能后来被多款软件重复需要时，先做影响分析，再提升为新的完整能力包；单款软件需求不能直接污染共享平台。

## 前台、后台和源码

### 用户前台可以有多套 UI

能力与 UI 模板分开选择。相同能力包可以提供标准 Web/桌面风格、紧凑工具风格、移动风格或后续品牌模板；模板不是只换颜色的皮肤，而是包含前台 Shell、布局、导航、主题、公共页面编排和扩展槽的一整套呈现方案。所有模板仍共用同一业务契约、状态机、SDK、错误语义和安全规则。第一阶段先把一套标准 UI 做完整，不能用大量皮肤掩盖功能未闭环。

UI 模板的选择记录在 Product Blueprint 中，由统一管理后台的创建向导完成；已装配软件更换模板必须先生成兼容、文件差异、冲突、测试和回滚计划，不能把模板切换实现成绕过 lock 的即时覆盖。模板负责呈现所选能力包的公共 Feature Block，不自行声明登录、会员或个人中心已经具备真实后端能力。

用户前台支持按页面混合三种交付方式：

- **Hosted UI**：平台托管页面，所有产品自动获得集中升级。
- **版本化组件依赖**：软件安装公共组件与 SDK，源码可查，按版本升级。
- **Generated Source**：把页面组合、路由、入口、主题和接入适配源码生成到软件仓库，允许维护和扩展。

禁止的是每款软件复制并私自改写登录、支付、权益等公共状态机；不是禁止交付源码。生成器只管理明确标记的 generated 区域，不能覆盖软件的 custom 代码。需要彻底修改公共实现时可以 `eject` 为产品自有分支，但必须记录已脱离自动升级并提供差异报告。

### 管理后台只有一套 UI

平台只维护一套统一管理后台。顶部选择“全部软件”或某一软件，左侧根据该产品已启用且真实可用的能力包注册目录，右侧显示对应管理功能。平台管理员、产品管理员和代理管理员按服务端权限范围看到不同内容。

新软件不会复制一套管理后台进自己的仓库。勾选能力包后，该包的管理 Feature Block 在统一后台中自动启用；关闭后入口消失且 API 同时拒绝新操作，历史数据按保留规则处理。

## 关键业务概念

- **User**：全局真实账号，可登录多款软件；账号安全状态全局共享。
- **Product**：一款独立软件；业务数据以 `product_id` 隔离。
- **Product Application**：同一软件的 Web、桌面、H5、小程序或 App 客户端表面与渠道。
- **Product Tenant**：某款软件下的官方或代理经营范围；不能脱离 Product 存在。
- **Entitlement**：某用户在指定产品和租户内可以使用什么、到什么时候、允许多少设备。
- **Platform Capability**：`capability_id` 表示的底层原子服务能力。
- **Complete Capability Package**：`package_id` 表示的、创建软件时可勾选的完整交付单元。
- **Product Blueprint**：新软件选择的目标端、能力包、UI 模板、品牌、渠道与扩展配置。
- **Assembly Manifest / Generated Project Lock**：记录实际装配和生成版本，用于复现、升级、冲突检测与回滚。

## 可持续扩展的通用功能库

通用功能不限于登录、会员和支付。候选范围包括账号安全、用户中心、角色权限、权益、设备、激活码、商品套餐、订单、支付退款、软件更新、远程配置、通知客服、文件存储、AI Gateway、额度用量、API Key、审计、代理租户、分销、运营报表、私有部署以及以后出现的其他重复需求。

范围很广不等于同时开工。只有完成完整能力包并通过目标端验收后，功能才从“候选/开发中”进入“可勾选”。

## 产品创建与缺失能力处理

创建新软件时，系统必须先验证所选能力包、UI 模板、目标端、Provider 和版本是否兼容。依赖不完整时阻止生成并说明缺口，不能生成看似完整但不可运行的项目。

发现缺失功能时分两类：

- 多款软件会复用：作为独立平台能力变更，先建立完整能力包、契约和影响分析，再进入可选目录。
- 只有当前软件需要：作为软件独有扩展留在该软件仓库，通过公开 SDK、API、事件和扩展点接入。

标准新软件装配默认不修改共享平台代码；接入中发现公共能力不足时必须停止标准装配，转为公共能力开发，不能夹带修改已发布接口。

## 阶段目标

### P0：证明“选择即可得到完整功能”

- 产品蓝图、能力包目录、依赖解析、装配清单和生成锁定清单。
- 创建 Product、official Tenant、Product Application 和测试凭据。
- 平台管理员登录、最小权限与审计基础。
- `package.account` 与 `package.entitlement` 第一条完整能力链。
- Web/桌面标准 UI 的登录、个人中心和当前权益源码/组件。
- 第一版 SDK、生成的接入壳和一款装配验收软件；不开发其真实正文业务。
- 统一后台真实用户查询与权益管理。

P0 只有在“不手写重复功能即可创建并运行装配验收软件”后通过，不能以后台菜单、契约或演示页面代替。

### P1：授权和商业化基础

- `package.device-license`：设备、设备上限、撤销、激活码与兑换。
- `package.commerce`：套餐、订单、支付、退款、收银台和最终权益。
- 第一套标准 UI 覆盖完整购买闭环。

### P2：交付、AI 与运营能力

- 软件版本、更新、远程配置和通知。
- AI Gateway、逻辑模型、额度、用量计费、API Key 和对账。
- 文件存储、运营指标和所需的多端适配。

### P3：真实需求驱动的增长与部署

- 分销、佣金、提现、优惠券、发票、团队席位、海外支付和多币种。
- 代理白标、自定义域名和代理独立支付。
- 私有部署实例、签名许可证、升级、备份恢复和诊断。

P3 不得因为参考项目里存在某功能就提前实现。

## 核心验收标准

判断项目方向是否正确，以这一条作为最高产品验收标准：

> 创建一款新软件，选择目标端、任意一组已经标记为可用且彼此兼容的通用能力包，并选择一套兼容的用户前台 UI 后，平台是否能够在不重新开发这些通用功能的情况下，立即为该软件装配完整可运行的用户前台、统一后台管理内容、真实后端能力、SDK/API、配置、可维护源码、测试和使用说明；开发者是否只需要继续开发该软件独有的业务。

登录、会员和支付只是最常见的示例，不是平台能力的范围上限。账号安全、用户中心、设备、激活码、更新、存储、通知、AI 调用与计费、代理经营以及以后多款软件重复需要的功能，都必须按同一标准建设和验收。

只有答案为“是”，这些能力包才算完成，项目才真正实现了减少重复开发。对应工程验收底线是：

- 产品 A 不能读取产品 B 的业务数据。
- 同一产品的代理 A1 不能读取代理 A2 的数据。
- 禁用能力后用户前台、统一后台和 API 行为一致。
- 相同蓝图可以重复生成；升级或重新生成不覆盖软件独有代码。
- 新产品和新能力包接入后，已发布旧产品继续通过回归测试。
- 客户端提交的产品、租户、价格、支付结果和权限结果均不可信。

## 明确非目标

- 不用菜单数量、页面外壳、契约数量或演示数据冒充完整能力。
- 不为每款软件复制共享后端、管理后台或公共业务状态机。
- 不追求第一版同时覆盖所有能力包、所有 UI 风格和所有端。
- 不把统一底座变成承载所有软件独有业务的巨型项目。
- 不从旧项目复制核心代码、数据库模型或安全实现。
- 第一版不做微服务，不把 Redis 或内存 Map 当作最终业务事实。

## 决策原则

1. 先减少真实重复开发，再扩充后台菜单。
2. 一个可勾选项必须对应一个完整能力包。
3. 一个业务事实只有一个权威模块，共享后端集中实现一份。
4. 生成源码必须区分平台管理区与产品自有区，不能静默覆盖。
5. 产品、租户、管理员权限和客户端身份默认拒绝越权。
6. 外部回调默认会重复、延迟、乱序或失败。
7. 每次只完成一条纵向能力链，再扩展下一包或下一端。
8. 产品范围看本文，实施顺序看 `roadmap.md`，真实完成度看 `implementation-status.md`。
