# 新软件标准接入规范

本文用于每一款新软件接入统一后台。目标是让接入工作只增加产品配置和客户端适配，不破坏已经接入的软件。

## 核心原则

```text
标准接入 = 创建产品配置 + 签发独立客户端身份 + 启用能力 + 接入统一 SDK + 验收
标准接入 != 为新软件修改共享后台业务逻辑
```

如果现有能力已经满足需求，接入过程中不得修改共享模块、数据库公共结构或已发布 API。只有新软件确实需要平台尚不存在的公共能力时，才另立“平台能力变更”，先更新能力索引和契约，再实施共享后台修改。

## 两种工作必须分开

### A. 标准产品接入

允许变更：

- 后台新增一条 Product 及其官方租户。
- 为测试环境和生产环境分别签发客户端身份。
- 为该产品启用已有公共能力并填写产品级配置。
- 在新软件仓库中安装统一 SDK、编写界面适配和业务调用。
- 新增该产品的接入测试数据和测试报告。

禁止变更：

- 不得修改其他产品的配置、凭据、数据或发布记录。
- 不得在共享后台增加 `if 产品A`、`if 产品B` 一类产品名称分支。
- 不得复制登录、权益、支付等后端代码到新软件或另建一套接口。
- 不得绕过 SDK 直接拼接未公开的后台接口。
- 不得为了通过接入测试而降低产品、租户或管理员权限检查。

### B. 平台公共能力变更

出现以下任一情况时，必须停止“标准接入”，另开公共能力变更：

- 需要新增 API、修改字段含义或增加新的业务状态。
- 需要修改共享数据库结构、身份模型、权限模型或产品隔离规则。
- 需要新增支付、存储、通知等 Provider。
- 需要改变统一 SDK 的公开方法或错误语义。

公共能力变更必须先完成影响分析、契约更新、兼容方案、跨产品回归测试和回滚方案，不能夹带在某款软件的接入代码中上线。

## 标准接入步骤

1. 在后台创建 Product，生成稳定且唯一的 `product_code`。
2. 系统自动创建该产品的 `official` 官方租户。
3. 分别创建测试和生产客户端身份；凭据不能跨环境复用。
4. 在能力开关中只启用软件实际需要的公共能力。
5. 在新软件中安装固定版本的统一 SDK，禁止直接复制 SDK 源码。
6. 调用 SDK 初始化、登录和权益检查；后续按阶段接入设备、激活码、支付、更新等能力。
7. 在测试环境执行当前产品的接入测试和跨产品隔离测试。
8. 使用灰度或受控范围发布，观察错误率、拒绝日志和旧产品健康状态。
9. 验收通过后记录产品、SDK 版本、启用能力、测试报告和回滚点。

## AI 接入任务的固定输入

每次让 AI 接入新软件，必须明确提供：

```text
新软件代码根目录：
统一后台代码根目录：只读还是允许修改
product_code：
目标环境：test / production
需要启用的能力：
统一 SDK 版本：
允许修改的文件范围：
禁止修改的文件范围：
验收测试：
```

默认规则是“统一后台代码只读”。如果 AI 判断必须修改后台，它应停止接入并说明缺失的公共能力、影响的旧客户端和建议契约，不能自行扩大修改范围。

## 客户端接口兼容规则

- 客户端 API 使用明确主版本，例如 `/api/v1`；同一主版本内保持向后兼容。
- 已发布字段不得改名、删除、改变类型或改变原有含义。
- 新增请求字段默认可选；新增响应字段不得要求旧客户端理解后才能继续工作。
- 新增枚举值时，SDK 必须能以 `unknown` 或安全默认状态处理。
- 错误码含义保持稳定；不得把原来的可重试错误静默改为不可重试错误。
- 已发布 SDK 版本不得覆盖重发；新版本使用新的语义版本号。
- 删除旧 API 前必须记录废弃时间、替代接口、迁移说明和受影响产品，并至少保留一个明确迁移周期。
- 数据库迁移只向前追加；已在任何环境执行过的迁移不得改写。

详细约束见 `platform/contracts/client-api-compatibility.md`。

## 产品与凭据隔离

- `product_id` 只能来自服务端认证后的客户端身份，不能相信请求正文或查询参数。
- `tenant_id` 由服务端根据官方渠道或代理绑定解析，不能让客户端任意切换。
- 测试和生产使用不同 `client_id`、密钥、数据库配置和回调地址。
- 后台日志可记录产品代码和请求追踪号，不记录完整令牌、密钥或用户密码。
- 一款软件的凭据泄漏时，只撤销该客户端身份，不影响其他产品。

## 接入验收清单

- 新软件可以建立客户端会话并得到正确 ProductContext。
- 正确用户可以登录，错误或冻结账号被拒绝。
- 权益只在当前 `product_id + tenant_id` 范围内生效。
- 新产品凭据无法读取任何旧产品数据。
- 旧产品凭据无法进入新产品上下文。
- 新产品关闭的能力既不显示入口，API 也明确拒绝。
- 网络失败、超时、令牌过期和服务端限流都有明确恢复路径。
- 当前支持的旧 SDK 契约测试全部通过。
- 至少一款已接入旧软件完成关键流程回归。
- 测试报告和回滚点已保存，且不包含真实密钥和用户数据。

以上项目全部满足，才能把新软件标记为“已接入生产环境”。

## 推荐的 AI 接入指令

```text
把这款软件接入统一后台。统一后台默认只读，只允许在新软件仓库内修改代码。
先读取新软件标准接入规范、SDK 文档和 OpenAPI 契约，使用指定 SDK 版本。
不得修改已发布接口、共享数据库结构或其他产品配置。
如果缺少公共能力，停止接入并提交影响分析，不要自行改后台。
完成后执行本产品接入测试、跨产品隔离测试和旧产品回归测试，并列出所有未验证项。
```
