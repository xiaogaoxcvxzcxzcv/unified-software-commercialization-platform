# 新软件创建、装配与旧软件接入规范

本文约束新软件从蓝图生成，以及已有软件接入统一底座。目标是让软件直接获得所选完整能力，而不是只拿到 API 后继续手写通用前台；同时保证新产品不会破坏已经接入的软件。

## 三种工作必须分开

### A. 新软件标准装配

```text
Product Blueprint
-> 选择目标端、available 能力包和 UI 模板
-> 解析依赖和配置
-> 自动创建 Product / official Tenant / Application / 凭据
-> 启用共享后端与统一管理后台功能
-> 生成用户前台组合、路由、接入壳、配置和 lock
-> 运行接入、隔离和回归测试
```

标准装配允许新增蓝图、产品配置、独立凭据和受控生成物；只允许使用最小测试夹具验证 custom 与 Extension 边界，不在装配任务中开发软件正文。共享后端、统一管理后台和公共 SDK 默认只读。软件独有业务和扩展在装配验收完成后进入该软件自己的独立开发任务。

### B. 已有软件标准接入

先建立软件基线和 Product Blueprint，再选择 Hosted UI、版本化组件或 Generated Source 逐步替换重复实现。不得直接删除原登录、支付或授权流程；迁移完成并通过兼容测试后再走废弃策略。

### C. 平台公共能力变更

出现以下任一情况时，停止标准装配，另立公共能力变更：

- 需要新增 API、业务状态、Provider、共享数据库结构或权限语义。
- 现有能力包缺少用户前台、统一后台、SDK、配置、源码或测试。
- 目标端或 UI 模板尚未达到 `available`。
- 需要修改公共 SDK、生成器或已发布兼容契约。

公共能力变更先完成 Manifest、影响分析、契约、兼容、跨产品回归和回滚，不能夹带在某款软件接入中上线。

## 新软件标准装配步骤

1. 创建 Product Blueprint，填写软件资料、品牌和合规文案。
2. 选择 Web、桌面、H5、小程序或 App，建立对应 Product Application。
3. 只选择 `capability-package-catalog.md` 中对目标端为 `available` 的完整能力包。
4. 选择兼容的 UI Template 和每个页面的 Hosted、Package 或 Generated Source 交付方式。
5. 补齐 Provider、渠道、回跳和非秘密配置；密钥只登记服务端引用。
6. 审阅 Assembly Plan，包括依赖、创建项、生成文件、迁移、测试和风险。
7. 执行装配，自动建立 official Tenant、环境隔离凭据、能力配置、统一后台注册和软件侧生成物。
8. 在生成软件根目录写入 `AGENTS.md` 和 `docs/software-development-handoff.md`，说明已选公共能力、源码所有权、扩展点、SDK/API/事件、命令和禁止事项。
9. 运行安装启动、所选公共能力黄金流程、产品/租户隔离、失败恢复、重生成和旧产品回归；需要验证扩展边界时只注入最小测试夹具。
10. 验收后保存 Assembly Manifest、Generated Project Lock、SDK/模板版本、测试报告和回滚点，并将生成软件移交给独立的软件正文开发任务。

## 生成物与源码边界

- 公共后端与唯一管理后台不复制到普通软件仓库。
- 新软件应得到可运行的前台 Shell、布局、导航、主题、所选公共页面组合、路由、接入适配、配置模板、示例和扩展源码。
- 平台不生成统一业务首页、业务目录页、工作台或核心内容；开发人员在 custom 区域实现这些独有业务，并通过登记的路由、导航、槽位、SDK/API 和事件接入。
- 公共 SDK 与 Feature Block 核心采用版本化依赖并提供可查源码，禁止手工复制成未登记分叉。
- 生成器只能更新 lock 记录的 generated 区域，禁止覆盖 custom 区域和未知文件。
- 需要修改公共实现时执行显式 eject；eject 后记录 forked，不再承诺自动覆盖升级。

## 装配完成后的独立软件开发

装配完成与软件正文完成是两个不同里程碑：

```text
统一底座装配任务
-> 公共前台/后台/后端/SDK 可运行且证据、交接齐全
-> 装配任务结束
-> 该软件自己的开发任务读取根目录 AGENTS.md 和交接说明
-> 只在 custom 与已登记 Extension 范围开发正文、目录、工作台和核心业务
```

统一底座不负责继续编写这款软件的真实正文内容。它只负责公共能力契约、升级、兼容、回滚和扩展接口。软件开发 AI 不得重新实现已选公共能力；需要调用公共能力时使用交接说明中登记的 SDK/API/事件和 Feature Block，需要新增独有能力时使用 custom 与 Extension Manifest。发现多款软件都会复用且现有包不足时，停止对共享层的直接修改，另立公共能力变更。

主计划中的装配验收软件 A/B/C 只用来证明创建、隔离、晋级和回归。A 可以携带一个最小扩展测试夹具；B 和 C 不需要开发业务正文，也不以页面数量或真实行业功能作为验收条件。

## 禁止事项

- 不得修改其他产品的配置、凭据、数据、生成锁或发布记录。
- 不得增加 `if 产品A`、`if 产品B` 一类产品名称分支。
- 不得复制登录、权益、支付等公共后端或状态机另建一套接口。
- 不得绕过 SDK/API Client 调用未公开接口或直接访问数据库。
- 不得把 planned、contracted、demo、implemented 或 verified 包显示为普通可选项。
- 不得为了通过接入测试降低产品、租户、能力、管理员或生成路径检查。

## AI 装配任务固定输入

```text
新软件代码根目录：
统一底座代码根目录：默认只读
Product Blueprint / product_code：
目标端与 Product Application：
目标环境：test / production
选择的 package_id + version：
UI Template + delivery mode：
SDK / generator 版本：
允许修改的 generated / integration 范围：
禁止修改的 custom / shared 范围：
验收测试与旧产品回归基线：
```

AI 判断已有包、模板或 SDK 不足时，必须停止标准装配并提交缺失交付面、兼容影响和公共能力建议，不能自行扩大共享代码修改范围。

## 接入与装配验收

- 蓝图依赖、目标端、模板、Provider 和版本兼容检查通过。
- 新软件获得清晰的 Assembly Manifest、交付清单和 Generated Project Lock。
- 新软件根目录包含可被后续 AI 自动发现的 `AGENTS.md` 和开发交接说明，并可据此找到允许修改范围、公共能力调用方式、扩展点及启动/测试入口。
- 生成项目可以安装、启动，并运行所选能力的真实用户前台。
- 统一后台只显示该产品已启用且可用的管理功能，并使用真实 API。
- 正确用户能够完成黄金流程；错误、冻结、超时、重试和撤销有恢复路径。
- 新产品凭据无法读取旧产品数据，旧产品凭据无法进入新产品上下文。
- 同产品代理租户之间隔离，软件关闭能力后前台、后台和 API 一致拒绝。
- 相同蓝图重复装配不重复创建事实；升级与重新生成不覆盖 custom 代码。
- 当前支持的旧 SDK、模板和至少一款旧产品关键流程回归通过。
- 报告和锁定清单不包含真实密钥、密码或用户数据。

以上全部满足，才能把一次装配标记为生产可交付。

## 推荐的 AI 指令

```text
按 Product Blueprint 把这款软件装配到统一能力底座。共享平台默认只读；先检查完整能力包目录、Assembly Manifest、UI 模板、SDK 和兼容契约。只修改装配计划登记的 generated/integration 范围，不编写软件正文，不覆盖 custom 或未知文件；扩展边界只使用明确登记的最小测试夹具。如果所选包或目标端尚未 available，停止并提交缺口，不要用演示页面或手写替代。必须在生成软件根目录交付 AGENTS.md 和开发交接说明。完成后执行所选公共能力黄金流程、隔离、重生成、升级回滚和旧产品回归，并列出所有未验证项，然后结束装配任务。
```

装配完成后，软件正文开发 AI 使用以下固定边界：

```text
先读取生成软件根目录 AGENTS.md、docs/software-development-handoff.md、Assembly Manifest 和 Generated Project Lock。登录、会员、权益及其他已选公共能力直接使用登记的 Feature Block、SDK/API 和事件，不重新实现。正文首页、目录、工作台、核心流程和独有数据只写入 custom 或 Extension Manifest 明确拥有的范围；不得修改 generated、未授权 integration、共享平台、其他模块表或 Repository。公共能力不足时停止并提交缺口与影响分析。
```
