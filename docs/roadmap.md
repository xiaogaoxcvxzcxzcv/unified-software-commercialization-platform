# 落地路线图

路线图按可验收结果推进，不以“生成了多少代码”判断进度。后续阶段开始前，前一阶段的契约和冒烟测试必须通过。

## 阶段 0：工程地基

目标：所有开发者和 AI 对项目边界、模块、契约和完成标准理解一致。

验收：

- 项目地图、治理规则、能力索引、ADR 和冒烟测试存在。
- 正式代码区与旧参考项目隔离。
- 产品、产品内租户、账号、权益契约完成初审。
- Product Application、三类能力术语、外部身份、商业流程编排和私有部署许可证完成初审。
- Git 和 CI 的基本检查可运行。

## 阶段 1：平台最小核心

目标：创建产品及其官方/代理租户，配置产品能力，登录用户并正确判断当前范围的权益。

范围：

- 后端启动、配置、迁移和健康检查。
- product、product-application、tenant、identity、access-control、entitlement、audit 模块。
- 管理员最小权限和审计。
- 管理后台登录、软件切换、产品能力目录、代理租户管理、用户查询、权益授予。
- 按 `phase-1-admin-navigation.md` 实现“全部软件”和“单款软件”两套不混杂的左侧目录。
- OpenAPI 与第一版 SDK。
- Hosted Login/Account 与 Web/桌面用户前台基础组件。

退出条件：ST-001 至 ST-006、ST-013、ST-021、ST-022、ST-024、ST-025 自动化通过；产品 A 无法读取产品 B 数据，同一产品的代理 A1 无法读取代理 A2 数据。

## 阶段 2：授权与基础商业闭环

目标：软件能够安全激活、绑定设备，并完成套餐、订单、支付到权益的第一条真实商业闭环。

范围：device、license、catalog、order、payment、commerce process、设备上限、兑换幂等、微信支付验签、退款基础路径和 Outbox。第一版会员采用固定期限 + 用户主动续费，不提前声称自动订阅。

退出条件：ST-007 至 ST-010 通过，至少一款真实软件在测试环境完成“套餐卡 -> 订单 -> 微信支付 -> 权益”闭环。

## 阶段 3：AI Gateway 与用量计费

目标：所有软件通过统一入口安全调用快速变化的 AI 模型，并得到可解释、可对账的成本和销售用量。商业闭环完成前只允许管理员手工测试额度，不向真实用户售卖。

范围：Provider 适配、动态模型目录、逻辑模型路由、流式调用、版本化价格、额度预占与结算、用量流水、开发者 API Key、用户前台用量页和后台计费页。

退出条件：ST-017 至 ST-020 通过；新增模型不修改软件客户端；重复结算不重复扣费；历史账单不受新价格影响。

## 阶段 4：版本与远程配置

目标：每款软件独立发布、灰度、回滚和更新远程内容。

范围：release、config、安装包哈希、签名、公告、二维码和功能开关。

退出条件：ST-011 通过，产品间版本和配置完全隔离。

## 阶段 5：存储与运营

目标：按产品和用户管理文件、AI 用量、成本、额度与统计。

范围：storage、文件配额、通知和运营报表；AI 用量账本已在阶段 3 完成。

退出条件：配额并发测试、私有文件授权测试、成本对账和备份恢复 ST-012 通过。

## 阶段 6：可选增长能力

只有出现明确业务需求后才进入本阶段。

可选范围：优惠券、发票、分销推广、佣金结算、提现、多币种、代理白标、自定义域名和代理独立支付配置。

退出条件：每项能力有独立需求、风控与财务规则、数据模型评审、ADR 和可对账的测试证据。参考项目里存在某项功能，不构成立项理由。

## 每个阶段的固定节奏

```text
确认需求
-> 更新能力索引和契约
-> 写领域规则测试
-> 实现后端与适配器
-> 生成/更新 OpenAPI Client
-> 实现 Feature Block 和页面
-> 跑冒烟测试
-> 更新文档与交付记录
```

一次只让一个核心业务链路进入“进行中”。AI 可以并行处理独立测试和文档，但不能同时发明互相依赖的数据模型。
