# 可装配能力底座落地路线图

路线图按“新软件减少了多少重复开发”推进，不按代码量、菜单数或横向模块数量判断进度。每个阶段必须产生可运行的纵向结果；未达到退出条件时，不扩散到更多能力包和 UI。

本文件只保留阶段级方向和退出条件。逐项工作编号、前置关系、九个交付面、失败路径、证据和验收细则以 `end-to-end-development-plan.md` 为唯一执行清单；两者冲突时先服从 `product-scope.md` 和 accepted ADR，再同步修正文档。

## G0：产品重心与工程规则封口

目标：所有开发者和 AI 对唯一中心、完整能力包、源码交付和完成标准理解一致。

范围：

- 重写产品范围、路线图、工程地图和接入规范。
- 建立完整能力包目录、Product Blueprint、Assembly、UI Template、Generated Project Lock 和扩展标准。
- 新增 ADR-0010，替代 ADR-0004 中禁止源码交付的绝对表述。
- 能力索引与两套 Feature Block Catalog 能追踪到 `package_id`。
- 实施状态明确区分 planned、demo、implemented、verified 和 available。

退出条件：全文搜索不再存在“统一后台是产品中心”“勾选菜单就代表能力完成”“新软件必须手写通用 UI”等冲突思想；文档链接、术语、ADR 和 UTF-8 检查通过。

## G1：软件创建与装配骨架

目标：能够创建一个不含软件独有业务内容的基础软件蓝图，并安全、可重复地产生前台框架和装配结果。

F0-02 管理员认证浏览器 E2E 的三个 P1 已修复并完成补救复验，状态为 `verified_after_remediation`。F0-03、G1 全部关口、G2A-01 Account v1 范围封口和 G2A-02 Identity 用户域与迁移已通过。G2A-03 用户认证和账号 API 已通过本地真实 PostgreSQL 与 Full 18/18，但托管 CI 因本地 Git 上传连接重置仍待验证。`package.account` 仍仅为 `contracted`，普通与 experimental 运行目录仍为空。当前唯一严格关口仍为 G2A-03；当前仍没有 `verified` 或 `available` 完整能力包。

范围：

- 创建向导：基本资料、目标端/Application、环境、能力包、UI 模板、品牌、渠道和确认。
- 完整能力包与 UI Template 版本目录，只展示指定目标端为 `available` 的组合。
- 自动建立 Product、official Tenant、Application 和测试凭据。
- Assembly Plan、Assembly Manifest、Generated Project Lock 和交付清单。
- 生成页面组合、路由、接入壳、配置模板和明确 custom 扩展区。
- 受信 Generator/SDK 工具目录，公开 upgrade/eject/rollback API 和可信 Extension Catalog 基础。
- 平台管理员登录、最小权限与审计基础收口。

退出条件：管理员认证与托管 CI 通过；标准模板完成视觉 QA；创建向导、单款软件管理工作区、装配记录、受信工具、lifecycle API 和 Extension Catalog 基础达到验收；无 `available` 包时创建入口正确失败关闭。Product Blueprint 至少需要一个真实能力包，因此 G1 不再要求用空包或假包创建“空白软件”；第一次真实点击创建并同时得到前后台统一在 G2 完成。

## G2：第一条真实完整能力链

目标：证明平台已经开始真正减少新软件开发时间。

范围：

- 完成 `package.account` 和 `package.entitlement` 的九个交付面。
- 第一套 Web/桌面标准 UI：登录、个人中心和当前权益。
- 第一版 SDK 和可运行装配验收软件；只用一个最小 custom/Extension 夹具验证扩展边界，不开发该软件的真实“工作台”或正文。
- 统一后台真实用户查询、权益授予、延长、撤销与审计。
- 真实 PostgreSQL、无演示 Client、完整失败和恢复路径。

固定黄金链：

```text
管理员登录
-> 通过服务端授权的 experimental 入口创建装配验收软件 A 的蓝图
-> 选择 Web/桌面 + account/entitlement + 标准 UI
-> 选择服务端授权的 output_target_ref
-> 自动建立 official Tenant、Application 和测试凭据
-> 生成前台组合源码、接入壳、SDK 配置和 lock
-> 生成软件本地 AI/开发人员交接说明
-> 启动装配验收软件 A
-> 用户通过公共登录页真实登录
-> 统一后台查询用户并授予权益
-> 装配验收软件 A 通过 SDK 检查权益
-> 个人中心显示真实权益
-> 用最小测试夹具验证禁用、隔离、重生成、升级/回滚、扩展边界和旧产品回归
-> 使用同一候选组合创建装配验收软件 B 并回归 A
-> 将通过门槛的能力包和标准 UI 晋级 ordinary available
-> 移除 experimental 权限，从普通入口创建装配验收软件 C
-> 再次同时验收该软件后台、用户前台和账号/权益黄金流
```

退出条件：ST-027 至 ST-033 全部通过，不手写登录、个人中心和权益状态机即可跑通上述链路，并验证最小独有扩展夹具、升级/回滚和第二产品回归；晋级后还必须在无 experimental 权限的普通 `/create` 入口完成软件 C 终验。A/B/C 都只验证所选公共能力能否正确出现在用户前台和统一后台、能否隔离与回归，不要求也不允许把其真实正文开发算入 G2。此前不得声称“创建软件后已经有完整前后台”或“能力包已经可供普通创建”。

## G3：证明能力包机制可以复制

目标：用同一标准加入第二类能力，并证明接入新产品不修改共享核心。

范围：`package.device-license` 的设备、设备上限、撤销、激活码生成/兑换、用户前台、统一后台、SDK 和测试。

退出条件：创建采用 `package.device-license` 的新装配验收软件时，只新增蓝图、产品配置、生成物和必要的最小独有测试夹具；既有装配验收软件回归通过；不存在产品名称硬编码。这里仍不开发验收软件的真实正文业务。

## G4：UI 选择与多端交付

目标：证明“一个能力，多种前台 UI”可用且可维护。

范围：

- 至少标准 Web UI 和紧凑桌面 UI 两套真实模板，不只是换颜色。
- Hosted UI、版本化组件依赖和 Generated Source 可按页面混用。
- 模板兼容矩阵、生成源码所有权、升级冲突、eject 和回滚。
- 按真实需求增加 H5、小程序或 App 适配，不同时铺满所有端。

退出条件：切换模板不改变 API、事件、支付/权益结论和安全语义；生成源码可维护；模板升级不覆盖独有业务。

## G5：商业闭环能力包

目标：新软件勾选商业能力后直接获得可用的购买闭环。

范围：`package.commerce`，包括 catalog、order、payment、commerce、套餐卡、购买确认、收银台、支付结果、订单、退款与对账。第一版采用固定期限权益 + 用户主动续费。

退出条件：至少一款装配验收软件在测试环境完成“套餐 -> 订单 -> 微信支付 -> 最终权益”；重复回调、支付中断、退款和恢复均通过，且不需要重新开发购买页面；该验收不要求开发软件正文。

## G6：按重复需求逐包扩展

候选顺序：

1. `package.release-config`：版本、更新、公告和远程配置。
2. `package.ai-usage`：AI Gateway、逻辑模型、额度、用量、API Key 和对账。
3. `package.storage` 与 `package.notification`。
4. 产品内代理经营与真实需要的多端适配。
5. 分销、结算、白标、多币种和私有部署等真实需求轨道。

每一项仍按完整能力包推进，不能批量增加后台菜单或只完成后端。

## G7：云端生产上线与运维终验

目标：在约定发布范围功能完成后，让机器从干净云环境完成部署、迁移、验证、恢复和回滚，人工只保留不可替代的外部授权与最终上线批准。

开发和测试不是等到 G7 才开始部署：每条可运行纵向链都必须依次经过本机、GitHub 托管 CI 和云端测试环境。G7 负责使用生产约束最终收口域名/TLS、Secret Manager/KMS、监控告警、容量压测、备份恢复、安全检查、灰度发布、回滚和运维手册。

退出条件：ST-012、ST-040、ST-041、ST-042 通过；生产服务不依赖开发电脑；高危安全问题为零；机器能从空环境部署并在发布失败时恢复；人工只完成云账号/域名/正式密钥授权和最终上线确认。未通过前不得声称“已正式上线”。

## 当前暂停项

在 G2 通过前暂停：

- 新增更多后台业务菜单、仪表盘装饰和演示开关。
- 横向实现 AI、支付、私有部署、分销、结算等更多模块。
- 同时制作多套 UI 皮肤。
- 把契约、占位模块、演示数据或可点击页面称为可交付能力。
- 为某款软件夹带修改共享 API 和数据库。

现有管理员认证可以作为 G1 平台基础收口，但完成后应回到装配主链。

## 每个能力包的固定节奏

```text
确认重复需求和用户结果
-> 检查能力包目录与原子能力索引
-> 更新 Manifest、契约和 ADR（如需要）
-> 实现后端、迁移、权限和恢复
-> 实现统一管理 Feature Block
-> 实现用户 Feature Block 与目标 UI
-> 发布 SDK/API、配置和生成源码
-> 在装配验收软件运行完整装配 E2E
-> 跑隔离、升级、回滚与旧产品回归
-> 更新证据后才标记 available
```

一次只让一条核心纵向链进入进行中。AI 可以并行做独立审计和测试，但不能同时发明相互依赖的业务模型。
